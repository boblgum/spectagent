# Spectagent вЂ“ Github Spec Kit & Opencode Docker Image

A self-contained Docker image based on **Debian (stable-slim)** that bundles:

| Tool | Purpose |
|---|---|
| **uv** | Fast Python package & project manager |
| **Python** (latest stable, managed by uv) | Runtime |
| **git** (latest from Debian repos) | Version control |
| **opencode** (latest stable) | AI coding agent |
| **Bun** (latest stable) | JavaScript runtime / package runner (`bunx`) |
| **oh-my-opencode** | Agent harness for opencode вЂ” orchestrates specialist LLM agents |
| **Basic Memory** (sidecar) | Persistent MCP knowledge base вЂ” agents read/write notes across sessions |

## Quick start

```bash
# 1. First-time setup: creates .env, docker-compose.secrets.yml,
#    and builds the image.
make init

# 2. Edit docker/docker-compose.secrets.yml вЂ“ uncomment only the providers you use

# 3. Create the secret files you selected and paste your keys
echo -n "sk-ant-вЂ¦" > secrets/anthropic_api_key.txt && chmod 600 secrets/anthropic_api_key.txt

# 4. Edit config/git/.gitconfig with your identity

# 5. Rebuild so the image picks up any changes
make build

# 6. (Optional) Install oh-my-opencode inside the container
make omo-install

# 7. Start the persistent knowledge brain (MCP sidecar)
make brain-start

# 8. Seed the brain with project knowledge
make brain-ingest

# 9. Open a shell вЂ“ the container starts, you work, it removes itself on exit
make shell
```

Inside the container you have `uv`, `python`, `git`, `opencode`, `bun`/`bunx`, and `specify` available.

## oh-my-opencode

[oh-my-opencode](https://github.com/code-yeongyu/oh-my-opencode) is an agent harness that supercharges opencode with specialist LLM agents (Sisyphus, Prometheus, Oracle, вЂ¦), advanced LSP/AST tools, and multi-provider model routing.

### Setup

Run the installer interactively inside the container:

```bash
make omo-install
```

The installer verifies the opencode integration and prompts to authenticate immediately after.

To re-run the installer (e.g. after an opencode upgrade):

```bash
make omo-reconfigure   # removes the install sentinel and re-runs omo-install
```


### Provider priority

`Native (anthropic/, openai/, google/) > GitHub Copilot > OpenCode Zen > Z.ai Coding Plan`

### Authentication

After installation, authenticate each provider through opencode's auth flow:

```bash
make omo-auth   # runs  opencode auth login  inside the container
```


## Makefile targets

Run `make help` to list all available targets:

```
  build           Build (or rebuild) the Docker image
  rebuild         Rebuild the Docker image (alias for build)
  shell           Open a bash shell inside the container (ephemeral, removed on exit)
  opencode        Continue last opencode session (use 'make opencode-new' for a fresh one)
  opencode-new    Start a brand-new opencode session
  omo             Run oh-my-opencode CLI inside the container
  specify         Run specify (spec-kit) inside the container
  python          Run python inside the container
  uv              Run uv inside the container
  git             Run git inside the container
  omo-install     Interactively install oh-my-opencode inside the container, then verify and auth
  omo-auth        Run opencode auth login inside the container interactively
  omo-reconfigure Remove sentinel and re-run the interactive installer
  brain-start     Start the Basic Memory brain sidecar (background service)
  brain-stop      Stop the Basic Memory brain sidecar
  brain-status    Show brain container status and health
  brain-logs      Tail logs from the brain container
  brain-ingest    Seed / refresh all priority knowledge into the brain
  brain-sync      Same as brain-ingest вЂ“ use after completing a feature
  brain-reset     Wipe the brain data and config volumes (asks for confirmation)
  env             Create .env from template if it does not exist yet
  secrets         Create docker/docker-compose.secrets.yml from example and secrets/ dir
  reset           Remove all files/dirs generated by make init (asks for confirmation)
  init            Full first-time setup: env, secrets, build
```

Each tool target starts a **fresh, ephemeral container** (`--rm`) on demand and removes it automatically when the command finishes. No background service is needed.

**Tool targets** accept arguments directly (no `ARGS=` parameter needed):

```bash
make opencode-new --help
make specify check
make python -c 'import sys; print(sys.version)'
make git status
make uv --version
```

Or use the `ARGS` variable for more complex commands:

```bash
make opencode-new ARGS="--help"
make specify ARGS="check"
```

> `make opencode` resumes the last session (`opencode --continue`).  
> `make opencode-new` starts a fresh session.

## Volumes / mounts

All configuration and the workspace are **mounted from the host** вЂ“ nothing is baked into the image.

| Host path (default) | Container path | Mode |
|---|---|---|
| `./workspace` | `/workspace` | read-write |
| `./config/opencode` | `/root/.config/opencode` | read-write |
| `./config/git/.gitconfig` | `/root/.gitconfig` | read-only |
| `./data/brain` | `/app/data` (brain service) | read-write |
| `docker volume: brain-config` | `/app/.basic-memory` (brain service) | read-write |

Override the host paths via environment variables in `docker/.env`:

- `APP_DIR` вЂ“ workspace directory
- `OPENCODE_CONFIG_DIR` вЂ“ opencode config directory
- `GIT_CONFIG_FILE` вЂ“ git global config file
- `BRAIN_DATA_DIR` вЂ“ brain knowledge notes directory (default: `../data/brain`)
- `BRAIN_PORT` вЂ“ host port for the brain HTTP/MCP endpoint (default: `8765`)

## Secrets (API keys)

Secrets are **user-defined** вЂ” not everyone uses the same providers.
The base `docker/docker-compose.yml` contains no secrets at all.
Each user creates their own `docker/docker-compose.secrets.yml` that declares
only the keys they need.

### First-time setup

```bash
# 1. Copy the template (done automatically by  make init)
cp docker/docker-compose.secrets.yml.example docker/docker-compose.secrets.yml

# 2. Edit docker/docker-compose.secrets.yml:
#    - Uncomment the providers you use
#    - Remove or comment out the ones you don't

# 3. Create the matching secret files
echo -n "sk-ant-вЂ¦" > secrets/anthropic_api_key.txt
chmod 600 secrets/anthropic_api_key.txt
```

### Adding a new provider

To add a secret that isn't in the template (e.g. Mistral):

1. Create the file: `echo -n "keyвЂ¦" > secrets/mistral_api_key.txt && chmod 600 secrets/mistral_api_key.txt`
2. Add it to `docker/docker-compose.secrets.yml`:
   ```yaml
   services:
     spectagent:
       secrets:
         - mistral_api_key        # в†ђ add here

   secrets:
     mistral_api_key:
       file: ../secrets/mistral_api_key.txt   # в†ђ and here
   ```
3. `make restart`

The entrypoint automatically exports it as `MISTRAL_API_KEY`.

### How it works

1. The Makefile merges both compose files: `-f docker/docker-compose.yml -f docker/docker-compose.secrets.yml`.
2. Docker Compose mounts each declared secret as a read-only file on **tmpfs** at `/run/secrets/`.
3. `entrypoint.sh` discovers every file in `/run/secrets/`, sanitises the name, and exports an upper-cased env var.
4. A blocklist prevents overwriting critical vars (`PATH`, `LD_PRELOAD`, вЂ¦).

### Security properties

- Secrets live on **tmpfs** вЂ” RAM-backed, never written to the container filesystem.
- Keys do **not** appear in `docker inspect`, `docker compose config`, or daemon logs.
- `docker/docker-compose.secrets.yml` and `secrets/*.txt` are git-ignored.
- File permissions default to `chmod 600` (owner-only).
- The entrypoint rejects filenames that would overwrite security-critical env vars.

## Basic Memory brain (persistent knowledge base)

The brain is a [Basic Memory](https://github.com/basicmachines-co/basic-memory) sidecar container that gives AI agents persistent, searchable knowledge across sessions via MCP.

### Architecture

```
opencode (spectagent container)
    в”‚  MCP SSE  в†’  http://brain:8000/sse
    в–ј
basic-memory (brain container)          в†ђ always-on background service
    в”‚
    в”њв”Ђв”Ђ /app/data  в†’  data/brain/       в†ђ host-visible markdown notes
    в””в”Ђв”Ђ /app/.basic-memory              в†ђ docker named volume (SQLite index)
```

opencode is wired to the brain automatically via `config/opencode/opencode.json` (`mcp.brain`). Agents can call `write_note`, `read_note`, `search_notes`, and other MCP tools without any extra setup.

### Workflow

```bash
# Start once (survives container restarts)
make brain-start

# Seed with project knowledge (run after make init, and after each feature)
make brain-ingest   # or: make brain-sync

# Check health
make brain-status

# Tail logs
make brain-logs

# Wipe everything and start fresh
make brain-reset
```

### What gets ingested

| Tier | Content | Brain path |
|---|---|---|
| 1 | `workspace/AGENTS.md` вЂ“ pipeline, conventions, anti-patterns | `spectagent/agents-knowledge` |
| 1 | `workspace/README.md` вЂ“ command reference | `spectagent/readme` |
| 1 | `workspace/.specify/memory/constitution.md` (per project) | `constitutions/<feature>` |
| 2 | `workspace/specs/<NNN>/spec.md`, `plan.md`, `data-model.md`, `research.md` | `specs/<feature>/<artifact>` |
| 2 | `workspace/specs/<NNN>/contracts/*.md` | `specs/<feature>/contracts/<name>` |
| 4 | `config/opencode/opencode.json` вЂ“ provider/model catalogue | `infrastructure/model-catalogue` |
| 4 | `docker/Dockerfile`, `entrypoint.sh`, `docker-compose.yml` | `infrastructure/*` |

### MCP endpoint

The brain is reachable from the host at `http://localhost:8765/sse` (or `$BRAIN_PORT`).
Inside the Docker network it is `http://brain:8000/sse`.
This allows other tools (Claude Desktop, Cursor, etc.) to connect to the same brain.

## Running without Compose

When running outside Compose, mount your secret files to `/run/secrets/`
so the entrypoint can pick them up:

```bash
docker run -it --rm \
  -v "$(pwd)/workspace:/workspace" \
  -v "$(pwd)/config/opencode:/root/.config/opencode" \
  -v "$(pwd)/config/git/.gitconfig:/root/.gitconfig:ro" \
  -v "$(pwd)/secrets/anthropic_api_key.txt:/run/secrets/anthropic_api_key:ro" \
  spectagent:latest
```

Add one `-v вЂ¦:/run/secrets/<name>:ro` flag per secret you need.
